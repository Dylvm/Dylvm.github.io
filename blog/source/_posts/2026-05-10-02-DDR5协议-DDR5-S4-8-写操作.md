## 4.8.0 写操作与读操作：对称中的不对称

在上一节 [DDR5-S4.7-读操作] 中，我们花了很长的篇幅讨论"RD 命令发出后，数据什么时候到"。写操作是读操作的镜像——Controller 把数据发送给 DRAM，而不是 DRAM 把数据返回给 Controller。但镜像并不意味着完全相同。写操作有自己独特的参数、独特的约束和独特的复杂性。

最直观的差异是延迟。**Write Latency（WL）** 定义为从 Write 命令的最后一个周期到 DQS 上第一个有效的数据采样沿（不包括 Write Preamble 的翻转）之间的时间。WL = CWL + AL，而 CWL（CAS Write Latency）通常比 CL（CAS Latency）短几个 tCK。以 DDR5-4800 为例，CL ≈ 34 tCK，而 CWL ≈ 29 tCK。为什么写比读快？因为写数据是 Host 主动驱动到 DQ 上的——Host 在发出 WR 命令后只需要等 CWL 个周期，就可以开始翻转 DQS 并驱动 DQ。它不需要等待 DRAM 内部的 Column Decoder 去选通 Sense Amplifier 的输出——那个延迟只存在于读路径上。

另一个根本性的差异是**谁驱动总线**。读操作中，DQS 和 DQ 由 DRAM 驱动（Host 接收）；写操作中，DQS 和 DQ 由 Host 驱动（DRAM 接收）。这个"方向反转"意味着写操作的 DQS Preamble 不是由 DRAM 产生，而是由 Host 的 PHY 产生——Host 必须精确地在 CWL 时刻开始翻转 DQS。如果 Host 的 DQS 时序不对（偏早或偏晚），DRAM 就无法在正确的时刻采样 DQ 数据。这就是为什么 Write Leveling 训练（[DDR5-S4.21-写均衡训练]）如此重要——它确保了 Host 的 DQS 到达 DRAM 时与 CK 对齐。

让我们从写操作最基本的时序开始，一步步深入。

## 4.8.1 写 Burst 的基本时序：从 WR 命令到最后一个数据拍

Figure 49 展示了 BL16 写操作的完整时序。和读操作一样，横轴是 CK 周期，纵轴上是 CK、CMD、DQS、DQ 四条信号线。但注意一个关键区别：DQS 和 DQ 是由 **Host（Controller）** 驱动的，不是 DRAM。

### 4.8.1.1 BL16 模式的时间线走读

让我们沿时间线从头走一遍：

**T0 到 Ta（命令发出前）**。CA 总线上是 DES 命令。DQS 保持静态，DQ 处于 Don't Care 状态（Note 3 原文："The DQ signal is shown as 'Don't Care' before the first Write data bit indicating DFE is disabled"）。这个"Don't Care"意味着**在 DFE 被禁用的情况下**，Host 不需要在写数据之前驱动 DQ——DQ 可以是任意电平。

但如果你使能了 DFE（[DDR5-S4.31-判决反馈均衡]），情况就变了——DQ 必须在第一个写数据 bit 之前**保持高电平至少 4 个 UI**。这 4 个 UI 的高电平是给 DRAM 内部的 DFE 自适应引擎提供"同步时间"——DFE 需要看到几个确定的数据 bit 来初始化它的 Tap 延迟线和系数。

**Ta 时刻：WR 命令的第 1 个 CK 周期**。CS_n 拉低，CA 总线承载 WR 命令编码——目标和 RD 命令相同：Bank Group、Bank、起始列地址。命令解码器识别出"这是一条写命令"。

**Ta+1 时刻：WR 命令的第 2 个 CK 周期**。CS_n 拉高，CA 总线承载第二段信息：列地址高位、Burst Length 选择位 BL\*、Auto-Precharge 位 AP、以及 **WR_Partial (WR_P)** 位（CA12）。**从这一刻起，CWL 的计时开始**。

**从 Ta+2 到 Ta+CWL-1（等待期）**。与读操作的"等待期"不同，写操作的等待期是 Host 的"准备时间"——Host PHY 在内部完成写数据的并转串（从 128-bit 内部总线到 8-bit DQ）、计算 DBI（如果使能）、计算 CRC（如果使能）。DRAM 在这段时间也会准备好接收——内部 Column Decoder 被预先配置到目标列。

**Ta+CWL 时刻：Host 开始驱动 DQS Preamble**。对于 Figure 49 的配置（Preamble = 2tCK "0010" Pattern），Host 先将 DQS_t 保持低电平 1 个 tCK，然后在第 2 个 tCK 做一次上升沿翻转。这个"0010"序列告诉 DRAM："写数据即将到达，请准备接收"。

**Ta+CWL+tWPRE 时刻：第一个数据拍 D0 被 Host 驱动到 DQ**。DQS 上升沿在 DQ 数据中心采样（图中是边沿对齐，但实际系统中 DQS 和 DQ 是中心对齐的，如 Note 所说"for clarity of illustration, DQS and DQ are shown center-aligned"）。DQS 每半个 CK 周期翻转一次，每个沿采样一个 DQ 数据拍：D0, D1, D2, ..., D15——共 16 拍，占 8 个 CK 周期。

**Ta+CWL+tWPRE+8tCK 时刻：最后一个数据拍 D15**。之后 DQS 进入 Postamble（tWPST）——再做一次翻转后回到静态电平。DRAM 在 Postamble 期间完成最后几个数据拍的锁存和 ECC 编码。

### 4.8.1.2 与读操作的关键对比

对比 Figure 35（读 BL16）和 Figure 49（写 BL16），你会发现：
- **命令发出后的"等待期"长度不同**：读等待期 = CL（34 tCK），写等待期 = CWL（29 tCK）。写比读少 5 tCK。
- **总线驱动者不同**：读是 DRAM 驱动，写是 Host 驱动。
- **Postamble 后数据去向不同**：读 Postamble 后 DQ 回到高阻（Host 停止驱动），写 Postamble 后 DRAM 内部开始 Write Recovery——数据从 SA 回写到存储电容（这是 tWR 的物理基础）。

### 4.8.1.3 BC8 写操作

与 BC8 读操作对称：写 BC8 只输出 8 个数据拍（4 个 CK 周期）。但 BC8 写操作有一个读操作没有的复杂性——**RMW（Read-Modify-Write）**。因为 DDR5 内部以 136-bit Code Word（128-bit 数据 + 8-bit ECC）为单位存储，只改写其中 64-bit 需要先读出整块、修改、重新计算 ECC、再写回去。这个 RMW 流程是 BC8 和 Partial Write 的核心约束——我们在 4.8.3 节中会深入讨论 WR_Partial 和 DM 如何触发 RMW。

## 4.8.2 写完数据之后：Precharge 和写恢复

写完数据并不等于操作结束。DRAM 把 Host 驱动的 DQ 数据锁存到了 Sense Amplifier 中——但这只是"暂存"。**SA 中的数据还必须回写到存储电容**——这就是 Write Recovery 的本质。

### 4.8.2.1 tWR：从最后一个写数据拍到 PRE 的安全距离

**tWR（Write Recovery Time）** 的参考点是**最后一个写数据拍之后的第一个 CK 上升沿**——不是最后一个数据拍本身。Figure 51 中 ta+10 是最后一个写数据拍 D15，ta+11 的 CK 上升沿是 tWR 计时的起点。从这个 CK 沿算起，再等 tWR 时间，Controller 才能发出 PRE 命令。

tWR 的物理原因非常直观：写数据到达 DRAM 的 I/O 引脚后，需要经过输入缓冲器 → Column Decoder → Sense Amplifier → Access Transistor → Storage Capacitor 这条路径。最后一步——给 Cs 充电——需要时间。如果 Controller 在 Cs 还没充完电的时候就发 PRE（关闭 WL），数据只写入了一半——电容上的电压处于一个模糊的中间状态，后续读出就会出错。

### 4.8.2.2 Figure 51 的完整走读

Figure 51 展示了 WRITE → PRECHARGE 的完整时序，同时对比了 BL16（上方波形）和 BC8（下方波形）两种模式。注意几个关键标注：

- `ta` 到 `ta+1`：WR 命令的两个周期
- `ta+CWL`：Host 开始驱动 DQS Preamble
- `ta+CWL+tWPRE`：第一个数据拍 D0
- `ta+10`（对于 BL16）：最后一个数据拍 D15
- `ta+11`：**tWR 计时的起点**——最后一个写数据拍之后的第一个 CK 上升沿
- `ta+11+tWR`：tWR 满足，可以发 PRE 命令
- `ta+11+tWR` 到 `ta+11+tWR+tRP`：PRE 到下一个 ACT 的间隔

在 BC8 波形中（下方），DQS 在 8 个数据拍后就停止翻转了，Postamble 更早开始——但这不会让 tWR 变短，因为内部 SA→Cs 的回写物理时间是不变的。

### 4.8.2.3 Auto-Precharge 写操作

Figure 52 展示了带 Auto-Precharge 的写操作（WRA 命令，CA11 = AP = L）。与 RD→PRE 的时序完全对称：Auto-Precharge 在写数据完成后自动执行 PRE，但内部要等到 tWR 满足后才真正开始 Precharge。Controller 不需要单独发 PRE 命令——在随机访问场景中，这减少了一条命令的开销。

## 4.8.3 为什么不是所有的写操作都一样：DM、Partial Write 和 RMW

在 4.8.1 中我们提到了一个细节——BC8 写操作可能触发 RMW。但为什么 BL16 的正常写操作不需要 RMW，而 BC8 和 Partial Write 需要？答案藏在 DDR5 内部存储的组织方式里。

### 4.8.3.1 136-bit Code Word 的约束

DDR5 内部以 **128-bit 数据 + 8-bit ECC = 136-bit Code Word** 为单位存储（[DDR5-S4.36-片上ECC]）。这个 136-bit 块是 ECC 编解码的最小单位——你不能只修改其中的一部分然后重新计算 ECC，因为 ECC 码是对整个 128-bit 数据块计算出来的。

当 Controller 发送一个 **完整的 BL16 写操作**（128-bit 数据，x8 器件）时，DRAM 收到完整的新数据 → 计算全新的 ECC → 将 136-bit 直接写入阵列。这是 **JW（Just Write）**——直接覆盖，不需要任何额外操作。

但当 Controller 只发送 **BC8**（64-bit 数据）或**使用 DM 屏蔽了部分数据拍**（Partial Write）时，DRAM 收到的不是完整的 128-bit——它必须**保留未写入部分的原始数据**。这就需要 **RMW（Read-Modify-Write）**：先读出该地址现有的 136-bit Code Word → ECC 解码 → 用新数据替换被修改的部分 → 重新计算 ECC → 写回 136-bit。

### 4.8.3.2 WR_Partial 位：Host 告诉 DRAM"这次写不完整"

DDR5 引入的 **WR_Partial（WR_P）**——Write 命令第 2 周期的 CA12——就是 Host 给 DRAM 的提示："这次写操作可能不完整，请准备好做 RMW"。

WR_P 与 DM 的协同关系是这样的：

- **WR_P = L 且 DM 使能**：Host 明确告诉 DRAM——"这不是全宽度写，你需要做 RMW"。DRAM 收到这个信号后，在写数据到达之前就启动了内部读操作（读出原 Code Word）。这节省了 RMW 的总延迟。
- **WR_P = H 且 DM 使能**：Host 说"这是全宽度写"，DM_n 必须全部为 H（无屏蔽）。DRAM 不做 RMW，直接覆盖。
- **WR_P = H 且 DM 禁用**：普通全宽度写。DRAM 直接覆盖。

**协议原文**（Section 4.8.1）写道："The WR_Partial (WR_P) = Low is to help DRAM start an internal read for 'read modify write' during masked writes." 关键词是 "help"——WR_P = L 不是必须的（即使 WR_P = H，DRAM 也能通过检测到 DM 来触发 RMW），但它给 DRAM 提供了**提前启动**内部读的机会，从而减少了 RMW 对性能的影响。

### 4.8.3.3 DM 的硬件细节

DM_n 是 DDR5 中**独立的输入引脚**（与 DDR4 中 DM/DBI 共用引脚的设计不同）。

在写操作中，DM Burst 的每一位与 DQ Burst 的对应位一一配对：DM Bit 0 对应 DQ Bit 0 所在的拍。DM = L 表示屏蔽该数据拍（该拍不被写入 SA），DM = H 表示正常写入。

**x4 器件不支持 DM**——没有 DM/DBI 引脚。x4 器件的 BC8 通过内部列地址控制来限制写入的列范围，不需要 DM。

## 4.8.4 写时序的核心参数：Figure 53 完整走读

如果说 Figure 49 是写操作的"宏观时间线"，那 Figure 53 就是写操作的"微观约束图"。它聚焦于单个写 Burst 内部的 DQS/DQ 时序关系，定义了 Controller PHY 必须满足的所有建立/保持时间参数。

### 4.8.4.1 图上的信号线

Figure 53 在纵轴上画了五组信号：CK 差分时钟、CA 总线上的命令（WRITE + DES）、三组 DQS/DQ/DM 波形对比。这三组波形展示了三种不同的时序关系：
- 第一组（最上方）：**tDQSS 的变化**——DQS 整体偏早、偏晚或刚好
- 第二组（中间）：**tDQoffset 的变化**——DQ 相对于 DQS 采样沿的建立/保持
- 第三组（最下方）：**DM 时序**——DM_n 的翻转必须在与对应 DQ bit 相同的窗口内

### 4.8.4.2 tDQSS：DQS 相对于 CK 的位置不是固定的

在 Write Leveling 训练完成之后，Host 的 DQS 被调整到相对于 DRAM CK 的一个特定相位——这个相位就是 **tDQSoffset**（训练锁定的值，范围 -0.5 ~ +0.5 tCK）。但训练不是一劳永逸的——温度和电压漂移会让 DQS 的实际到达时间在 tDQSoffset 的基础上继续漂移。

**tDQSS** 量化了这种系统级别的漂移。它有三级：
- **tDQSS.nom**：标称位置（训练后的理想值）
- **tDQSS.min**：DQS 最早可能到达的时刻（相对于 CWL 参考 CK 沿）
- **tDQSS.max**：DQS 最晚可能到达的时刻

Figure 53 的第一组波形展示了这三种情况下的 DQS-DQ 关系。当 DQS 偏早（tDQSS.min）时，DQ 相对于 DQS 的建立时间变短——数据可能还没稳定就被采样了。当 DQS 偏晚（tDQSS.max）时，保持时间变短——数据可能在下一次翻转前就被冲掉了。

### 4.8.4.3 tDQoffset：DQ 必须在 DQS 沿的什么窗口内

对于每个 DQS 采样沿（上升沿或下降沿），DQ 数据必须满足：
- **tDQoffset.min（建立时间）**：DQS 沿到来之前，DQ 必须已经稳定
- **tDQoffset.max（保持时间）**：DQS 沿之后，DQ 必须继续保持稳定

Figure 53 的第二组波形画出了 DINn（第 n 个写数据 bit）和 DINn+14（第 n+14 个 bit）的这两个约束——14 UI 的间隔意味着 DINn 和 DINn+14 在 Burst 中相隔 14 个数据拍。图中可以清楚地看到：每个 DIN 有自己的 tDQoffset.min 和 tDQoffset.max 窗口，DQS 沿必须落在窗口中间。

### 4.8.4.4 DM_n 的时序与 DQ 完全相同

DM_n 在写操作中用与 DQ 相同的时序和终端行为——DM 的建立时间和保持时间等于 DQ 的建立时间和保持时间。图中第三组波形的 DM_n2 展示了在 DQ2 的对应拍内 DM_n 的翻转——DM_n2 的沿与 DQ2 的沿在同一时刻被 DQS 采样。这确保了 DM 屏蔽的是正确的数据拍。

## 4.8.6 连续写操作的命令间隔

在 4.8.3 中我们建立了 JW vs RMW 的概念。这两个概念直接决定了同 BG 内连续写命令的间隔约束。

### 4.8.6.1 同 BG 内的写间隔：由 RMW 还是 JW 决定

Table 51 首先定义了哪些操作属于 RMW、哪些属于 JW：

- **x4 器件**：BL16 是 RMW（因为 x4 BL16 = 64-bit < 128-bit，不完整）；BL32 是 JW（128-bit，完整覆盖）
- **x8/x16 器件**：BL16 是 JW（完整的 128-bit）；BC8 是 RMW（只覆盖 64-bit）
- **x8/x16 器件，DM 使能**：所有 Partial Write 都是 RMW

RMW 操作的命令间隔（**tCCD_L_WR** 或 **tCCD_M_WR**）比 JW 操作的间隔（**tCCD_L_WR2**）更长——因为 RMW 多了内部读和 ECC 重新计算的时间。

Tables 52-54 详细列出了同 BG 同 Bank、同 BG 不同 Bank、同 BG 任意 Bank（JW）三种子情况的间隔约束。关键规则：**BL32 的 RMW 时序在 CRC 使能时额外增加 1nCK**（因为 CRC 位多占了一个 UI：8nCK + tCCD_L_WR 变成 9nCK + tCCD_L_WR）。

### 4.8.6.2 跨 BG 的写间隔：更简单

跨 BG 时 IO 通路独立，不需要考虑共享 IO 的冲突——连续写命令的间隔固定为 **8nCK**（BL16/BC8）或 **16nCK**（BL32）。这与读操作的跨 BG 规则一致——跨 BG 不区分 JW/RMW，因为 IO 通路的独立性消除了流水线冲突。

## 4.8.7 当写时序出错了：违例的分类和恢复手段

如果 Controller 的 DQ-DQS 时序不满足建立/保持要求，或者 DQS 本身的 Preamble/Postamble 超出了允许范围——数据可能被写错。JEDEC 在 Section 4.8.7 中明确定义了两种违例场景下的 DRAM 行为。这些规则的核心精神是：**错误不应该扩散——DRAM 不能因此"宕机"，并且有明确的恢复路径**。

### 4.8.7.1 数据眼图违例：影响只限当前地址

如果只是某个数据拍的 DQ 建立/保持时间不够（违反了 tRx_RDQ_tMargin、tRx_DQS2DQ_Skew 等参数）——DRAM 可能在该拍采到了错误的数据。结果就是：**只有这次 WRITE 命令覆盖的地址可能被写错**。DRAM 其他部分（其他 Bank、本 Bank 的其他 Row）完全不受影响。后续从该地址读取会返回"不可预测的数据"——但系统可以通过 CRC 检测到这个错误，然后重写正确的数据。

### 4.8.7.2 Strobe 违例：可能导致数据流水线错位

如果 DQS 本身的时序出问题了——tWPRE（Preamble 太短）、tWPST（Postamble 太短或太长）、tDQSS（DQS 到 CK 的偏移超出了范围）——问题就严重得多。因为 DQS 的边沿是 DRAM 内部数据流水线的"节拍器"：DRAM 用 DQS 的上升沿和下降沿来锁存 DQ 数据，并按照 Burst Order 把每个数据拍分配到正确的列地址。如果 DQS 的边沿数量和位置与预期不符，整个数据分配都可能错位。

**协议保证**：在满足以下条件时，DRAM 不会"宕机"（hang up）：
1. Write CRC 和数据 Burst OTF 都处于禁用状态
2. DQS（含 Preamble）出现在 CWL 参考位置 ±6 个 DQS 跳变沿的窗口内
3. 违规写之后，对其他已打开 Bank 的读命令可以正常工作
4. 违规写之后对同 Bank 的后续写可以继续发出——但数据也可能是错的
5. **一旦发出一条 PRE 命令**（任意类型——PREpb、PREsb 或 PREab），将所有 Bank 拉回 IDLE 状态 → 内部数据通路被完全重置 → **后续的非违规写可以正确写入数据**

**最重要的教训**：PRE 是"万能重置按钮"。无论写时序出了什么问题，只要把 Bank 预充电回 IDLE，内部状态机就被清零了——下一条写命令可以恢复正常。

## 4.8.8 DQS Preamble 的安全窗口：Write Enable Timings

在 [DDR5-S4.21-写均衡训练] 中，Write Leveling 训练确定了 Host DQS 相对于 DRAM CK 的最优相位。但训练完成后，由于**温度漂移**和**电压漂移**，DQS 的实际到达时间仍然会围绕训练值波动。Write Enable Timing（Section 4.8.8）量化了这种波动，并定义了 DRAM 内部"写使能窗口"必须覆盖的漂移范围。

### 4.8.8.1 三重漂移来源

**tDQSoffset**：Write Leveling 训练锁定的 DQS 偏移值。范围 -0.5 ~ +0.5 tCK。这是"有意偏移"——训练故意把 DQS 放到这个位置。

**tDQSS（系统级漂移）**：Host PHY 输出 + PCB 走线 + 连接器的总漂移量。Figure 60 展示了 CWL 参考 CK 沿之后，DQS 可能落在的"最早位置"（tDQSS.min）和"最晚位置"（tDQSS.max）。tDQSS 的漂移范围 = ±0.25 × tWPRE_EN_ntCK(min)。注意 Note 3：DDR5-3200 以下的数据速率，tDQSS 的绝对值还需要按速率比例缩放（例如 2000 MT/s 时，tDQSS(max) = (2000/2933) × 0.25 × tWPRE_EN_ntCK(min)）。

**tDQSD（DRAM 内部漂移）**：DRAM 芯片内部由于温度和电压变化引起的 DQS 到达时间变化。Figure 61 展示了这个概念——它与 tRX_DQS2DQ（[DDR5-S4.33-温漂偏移]）是同一类物理效应，都是内部 CMOS 门延迟随温度的漂移。

### 4.8.8.2 tWPRE_EN：使能窗口必须盖住所有漂移

DRAM 内部的 Write Preamble 检测逻辑有一个**使能窗口（Enable Window）**——在这个窗口内，DRAM 期望看到 DQS 开始翻转。这个窗口的大小就是 **tWPRE_EN_ntCK**。

使能窗口必须足够大，以覆盖所有漂移：

```
tWPRE_EN_ntCK ≥ |tDQSSmin| + tDQSSmax + |tDQSDmin| + tDQSDmax
```

如果使能窗口太小，DQS 的 Preamble 可能部分或全部落在窗口外面——DRAM 检测不到 Preamble，整个写 Burst 的数据都会丢失。

Table 56 列出了各 Preamble 长度对应的最小使能窗口：2tCK Preamble 需要 **1.5 tCK** 的使能窗口，3tCK 和 4tCK 各需要 **2.5 tCK**。

### 4.8.8.3 tWLS/H：Write Leveling 反馈的精度

Write Leveling 训练通过 DQ 反馈 DQS-vs-CK 的采样值。这个反馈本身的建立/保持时间就是 **tWLS/H = ±80 ps**。DDR5-6400 中 1 UI = 156 ps，80 ps ≈ 0.5 UI——训练精度约为半个数据拍。tDQSoffset 的最终值包含了这个 ±80 ps 的测量不确定性。

## 4.8.9 回顾与总结

写操作是读操作的镜像——但它有自己独特的约束和复杂性。我们从 WR 命令到最后一个数据拍的完整时间线开始，沿着"数据怎么进入 DRAM"这条路径，逐步理清了每一个时序参数背后的物理原因：

**写操作的核心时间线**：WR 命令 → CWL → Host 驱动 DQS Preamble → 数据拍 D0~D15（或 D31）→ tWR 写恢复 → PRE。读等待的是 DRAM 内部解码延迟（CL），写等待的是 Host 自身的 PHY 准备时间（CWL）——这就是 CWL < CL 的原因。

**DM 和 WR_Partial 存在的理由**：136-bit ECC Code Word 不能部分修改。当写操作只覆盖 128-bit 中的一部分时，DRAM 必须做 RMW——先读后改后写。WR_P = L 给 DRAM 提前启动内部读的机会，减少 RMW 的延迟惩罚。

**连续写操作的间隔取决于 JW 还是 RMW**：JW（全宽度覆盖）的间隔更短（tCCD_L_WR2），RMW（先读后改后写）的间隔更长（tCCD_L_WR）。跨 BG 的写操作间隔固定为 8nCK 或 16nCK——不受 RMW/JW 影响。

**Write Enable Timings 定义了从训练到运行的漂移容忍度**：Write Leveling 训练锁定了 DQS 的最优相位（tDQSoffset），但温度和电压漂移让 DQS 的实际到达时间继续变化（tDQSS + tDQSD）。tWPRE_EN 必须大于所有漂移的总和——否则 DRAM 检测不到 Preamble。

**写时序出错时的恢复路径**：PRE 命令将 Bank 拉回 IDLE，内部数据通路完全重置。这是万能的恢复手段——无论 DQS 时序怎么偏移，PRE 之后一切归零。
