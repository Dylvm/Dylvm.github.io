## 4.35.0 当 Controller 还不会"写"的时候

Write Leveling 训练的场景有一个根本性的矛盾：Controller 需要让 DRAM 执行写操作（发 WRITE 命令），才能触发 DRAM 内部的 Write Leveling Pulse。但此时 Controller 的 DQ-DQS 时序还没有对齐——它无法可靠地把实际数据驱动到 DQ 线上。如果强行用普通 WRITE 命令驱动 DQ，DRAM 可能采到垃圾数据——虽然 Write Leveling 的目的不是真正写数据（只关心 DQS 的相位），但误写入的数据可能破坏已经存储的内容。

**Write Pattern（WRP/WRPA）命令** 就是为解决这个矛盾而生的。WRP 和普通 WRITE 在**命令层面**完全相同——它们都触发 DRAM 内部的写操作流程（包括产生 Write Leveling Pulse）。唯一的区别是：**DRAM 不期望 Controller 驱动 DQ 线**——DRAM 用内部生成的固定 Pattern 代替 DQ 数据来"填充"写 Burst 的数据位。

---

## 4.35.1 WRP 在训练中的使用场景

在 Write Leveling 中，Controller 发 WRP 命令 → DRAM 产生内部 Write Leveling Pulse → Controller 发 DQS 脉冲去采样 → DQ 反馈。整个过程**Controller 不需要驱动 DQ 线**——DQ 线的电平完全由 DRAM 自己生成（虽然这些内部数据值本身没有意义，但 WL 训练只用 DQ 反馈相位、不关心数据值）。

WRP 也用于 Write DQ-DQS Deskew 训练中的**Loopback** 场景：Controller 发 WRP → DRAM 内部生成 Pattern 并写入 SA → Controller 发 RD 读回 → 比较发送和接收的差异 → 调整 per-bit 写延迟。在这个训练中，WRP 提供了"已知的写侧激励"——Controller 知道 DRAM 内部写入了什么 Pattern（通过 WRP Pattern 配置），然后读回来验证。

---

## 4.35.2 WRP/WRPA 的命令编码与配置

WRP 和 WRPA（Write Pattern with Auto-Precharge）是 **2-Cycle 命令**，在 Table 30 中有独立的编码。WRP 的 CA[5:2] = LHLL（区别于普通 WRITE 的 CA[5:2] = HHLL）。WRPA 增加了 Auto-Precharge——Pattern 写完后自动关闭 Bank。

WRP 支持 BL16 和 BL32 模式。内部使用的 Pattern 内容通过 **MR48（Write Pattern Mode）** 配置。
