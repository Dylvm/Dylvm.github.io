> 在 02 章中，我们看到了许多 tXXX 参数——tRCD、tCL、tRAS、tRP、tWR、tWTR、tRFC……本章将所有这些参数汇总，**统一解释每一个参数的物理来源、对应的 DRAM 内部操作、以及在不同代际中的典型数值**。

---

## 3.1 参数分类：Controller 约束 vs DRAM 特性

时序参数分为两类：

- **Controller 必须满足的约束（Txc）**：Controller 在发出下一个命令之前必须等待的时间——违反这些约束会导致 DRAM 误操作
- **DRAM 保证的特性（Txd）**：DRAM 内部操作完成所需的时间——Controller 不需要主动满足，但必须等这些时间过去才能发下一个命令

---

## 3.2 核心时序参数详解

### tRCD — RAS-to-CAS Delay（行到列延迟）

| 属性 | 值 |
|------|-----|
| **物理原因** | SA 完成 Access → Sense → Restore 三个阶段的时间 |
| **对应操作** | ACT 命令 → READ/WRITE 命令 |
| **DDR5-4800** | \~14 ns (34 tCK) |
| **DDR4-3200** | \~13.75 ns (22 tCK) |
| **各代趋势** | ns 值缓慢下降（模拟电路物理极限），但 tCK 数增加（tCK 缩短更快） |

### tCL — CAS Latency（CAS 延迟）

| 属性 | 值 |
|------|-----|
| **物理原因** | Column Decoder 译码 + CSL 选通 + I/O Gating 传输 + SERDES 串行化 |
| **对应操作** | READ 命令 → 第一个数据 bit 出现在 DQ |
| **DDR5-4800** | \~14 ns (34 tCK) |
| **DDR4-3200** | \~13.75 ns (22 tCK) |
| **各代趋势** | 与 tRCD 类似——ns 接近，tCK 数增加 |

### tCWL — CAS Write Latency（CAS 写延迟）

| 属性 | 值 |
|------|-----|
| **物理原因** | 类似 tCL 但对应写路径（Column Decoder 选通"哪些 SA 被覆盖"——不需要等待 SA 读出） |
| **对应操作** | WRITE 命令 → Controller 驱动第一个数据 bit 到 DQ |
| **特点** | 通常 **CWL = CL - 2**——写比读快 2 个 tCK |

### tRAS — Row Active Time（行激活时间）

| 属性 | 值 |
|------|-----|
| **物理原因** | SA Restore 完成（Phase 3）——SA 将满摆幅数据写回 Cs 的最短时间 |
| **对应操作** | ACT → PRE 的最小间隔 |
| **DDR5-4800** | \~32 ns (77 tCK) |
| **重要约束** | 即使你在 tRCD 之后立刻 READ 且 CL 很短、数据也很快读出了——**SA 回写还没完成**，不能 PRE。tRAS 是"最慢的那个环节" |

### tRP — Row Precharge Time（行预充电时间）

| 属性 | 值 |
|------|-----|
| **物理原因** | EQ 电路将 BL 和 /BL 短接并拉到 Vref(Vcc/2) 的时间 |
| **对应操作** | PRE → 下一个 ACT 的最小间隔 |
| **DDR5-4800** | \~14 ns (34 tCK) |

### tRC — Row Cycle Time（行周期时间）

| 属性 | 值 |
|------|-----|
| **物理原因** | 同一 Bank 完整操作周期——打开一行→操作→关闭 |
| **公式** | **tRC = tRAS + tRP** |
| **DDR5-4800** | \~46 ns (111 tCK) |
| **重要性** | **同一 Bank 连续两次 ACT 的最小间隔**——这是 Bank 交错需求的根本原因 |

### tWR — Write Recovery Time（写恢复时间）

| 属性 | 值 |
|------|-----|
| **物理原因** | 写数据通过 Access Transistor 对 Cs 充电（或放电）的时间 |
| **对应操作** | 最后一个写数据拍 → PRE 的最小间隔 |
| **DDR5-4800** | \~15-30 ns |

### tWTR — Write-to-Read Delay（写到读延迟）

| 属性 | 值 |
|------|-----|
| **物理原因** | 写数据从 DQ→I/O Gating→SA 的延迟——tWTR 之后**I/O Gating 被释放** |
| **与 tWR 的区别** | tWTR < tWR：I/O Gating→SA 快于 SA→Cs（电容充电） |
| **DDR5-4800** | \~2.5 ns (S) / 7.5 ns (L) |

### tRTP — Read-to-Precharge Delay（读到预充电延迟）

| 属性 | 值 |
|------|-----|
| **物理原因** | 最后一个读数据拍传输完毕 + SA 稳定 |
| **对应操作** | READ → PRE 的最小间隔 |
| **DDR5-4800** | \~7.5 ns (18 tCK) |

### tRFC — Refresh Cycle Time（刷新周期时间）

| 属性 | 值 |
|------|-----|
| **物理原因** | 全 Bank 刷新一行的时间（ACT → SA → Restore → PRE，不对外的全流程） |
| **DDR5-4800 16Gb** | \~295 ns (709 tCK) |
| **各代趋势** | **增加**——每一代密度翻倍 → 行数翻倍 → 刷新需要更长时间 |

### tREFI — Refresh Interval（刷新间隔）

| 属性 | 值 |
|------|-----|
| **物理原因** | 64ms 刷新窗口 / 需要刷新的总行数 / Bank 数的分配结果 |
| **DDR5-4800 1x** | **3.9 μs** |
| **DDR4-3200 1x** | **7.8 μs**——DDR5 减半是因为行数翻倍 |

### tRRD — Row-to-Row Delay（行到行延迟）

| 属性 | 值 |
|------|-----|
| **物理原因** | 不同 Bank 的 ACT 命令的最小间隔——限制峰值电流抽取 |
| **tRRD_S (不同 BG)** | 短间隔——不同 BG 的 Bank 切换更快 |
| **tRRD_L (同 BG)** | 长间隔——共享 IO 的 Bank 切换更慢 |

### tFAW — Four-Bank Activation Window（四 Bank 激活窗口）

| 属性 | 值 |
|------|-----|
| **物理原因** | 滚动时间窗口内不允许超过四次 ACT——**功耗限制** |
| **DDR5-4800** | \~21 ns (50 tCK) |

## 3.3 各代时序参数对比

| 参数 | SDRAM (66MHz) | DDR (133MHz) | DDR2 (400MHz) | DDR4-3200 | DDR5-4800 | 趋势 |
|------|-------------|-------------|--------------|-----------|-----------|------|
| tCK (ns) | 15 | 7.5 | 2.5 | 0.625 | 0.416 | ↓ |
| tRCD (ns) | 30 | 22.5 | 12.5 | 13.75 | 14 | → (\~14ns 是物理极限) |
| tCL (ns) | 30 | 22.5 | 12.5 | 13.75 | 14 | → |
| CL (tCK) | 2 | 3 | 5 | 22 | 34 | ↑ (tCK 缩短更快) |
| tRC (ns) | 90 | 60 | 55 | 46 | 46 | ↓ (缓慢改善) |
| tRFC (ns) | 90 | 75 | 127.5 | 350 | 295 | ↑ (密度翻倍) |
| tREFI (μs) | 15.6 | 7.8 | 7.8 | 7.8 | 3.9 | ↓ (行数增加) |

**核心规律**：**ns 值在趋近物理极限 → tCK 持续缩短 → tCK 数持续增加 → Controller 需要在更多的周期内管理更多的 in-flight 操作。**

## 3.4 本章总结与下章预告

每个 tXXX 参数背后都有明确的物理过程——没有一个是 JEDEC 委员会"随便定的"。tRCD = SA 四阶段的物理时间，tWR = Cs 充电时间，tRFC = 刷新一行的时间，tREFI = 64ms 窗口除以需要刷新的行数。

但问题来了：SA 的速度有上限（\~200-400 MHz，对应 tRCD 的物理极限），而市场要求接口从 100MHz 提速到 4800MHz——怎么调和？答案是**Prefetch**——内部以低频宽位宽工作，接口以高频窄位宽输出。下一章讨论 DDR 每一代提速的秘密武器。
