## 4.36.0 从"数据可能出错"到"错了也能纠正"

在 [DDR芯片存储原理-完整篇] 第 1 章中，我们讨论了 DRAM 电容的漏电问题——电荷逐渐流失，电压漂移，读出的 bit 可能翻转。DDR5 的密度比 DDR4 翻倍（16Gb → 32Gb, 64Gb），电容更小（~5-8 fF）、漏电更敏感。在 64Gb 的 die 上，即使刷新周期完全合规，随机单 bit 错误的概率也显著高于前几代。

传统方案是在 DIMM 上加一颗独立的 ECC 芯片（外部 ECC）——但这增加了成本和 PCB 面积。DDR5 把 ECC 直接集成到了**芯片内部**。内部阵列不再以 128-bit 为单位存储，而是以 **128-bit 数据 + 8-bit ECC = 136-bit Code Word** 为单位。这 8-bit 的 ECC 码基于汉明码变体，可以实现 **SEC（单纠错）**——检测并自动纠正 1-bit 错误——以及 **DED（双检错）**——检测（但不保证纠正）2-bit 错误。

对 Controller 来说，ECC 是**完全透明**的。写路径上：Controller 送 128-bit → DRAM 内部编码器计算 8-bit ECC → 136-bit 写入阵列。读路径上：阵列读出 136-bit → ECC 解码器纠错（如有 1-bit 错）→ 128-bit 返回 Controller。Controller 永远只看到 128-bit，不知道 ECC 的存在。

## 4.37 ECS（错误检查和擦洗）：DRAM 自主"巡逻"

ECC 可以纠正单 bit 错误——但如果不去主动发现和修复这些错误，它们会累积。一个 Code Word 中如果已经存在一个未纠正的 bit 错误（"脏 bit"），再出现第二个错误——ECC 就无法纠正了（只能检测）。

**ECS（Error Check and Scrub）** 就是 DRAM 内部的后台"巡逻"机制：DRAM 在空闲时自主扫描所有 Code Word，如果发现 1-bit 错误 → 纠正 → 将纠正后的数据写回阵列（擦洗）。这防止了单 bit 错误累积成多 bit 错误。

ECS 通过 **MR14 OP[2:1]** 配置为 Disable / Single Pass（一次性全扫描）/ Continuous（持续循环）。后台扫描占用 <1% 的内部带宽——几乎不影响正常读写性能。在 Self Refresh 期间，如果 MR15 使能了自动 ECS，DRAM 会在自刷新期间自主执行擦洗——零性能影响。

当错误数超过 **MR15** 配置的阈值（每 Gb 错误计数），DRAM 将错误最多的 Row 地址记录到 **MR16~MR19**，总错误计数记录到 **MR20**。Controller 可以定期通过 MRR 读取这些寄存器，监控存储阵列的"健康状态"——如果某一行频繁出错，Controller 可以主动触发 PPR（[DDR5-S4.29-封装后修复]）用冗余行替换它。
