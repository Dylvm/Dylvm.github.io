## 4.16.0 共享总线的困境：同一条命令，不同芯片需要不同配置

一条 DDR5 DIMM 上有 8~16 颗 DRAM 芯片，它们共享同一套 CA 总线。发一条 MRW 命令——所有芯片同时收到，所有芯片的同一个 Mode Register 被写到同一个值。这在配置"公共参数"（如 CL、BL）时很方便——一条命令搞定全部。

但麻烦在于：不同物理位置的芯片面临着完全不同的信号环境。最靠近 Controller 引脚的那颗芯片（DIMM 金手指附近）和 DIMM 最远端的那颗芯片，走线长度差了几厘米。这意味着 DQ 信号的到达时间和衰减不同（需要不同的 VrefDQ）、DQS 到达相位不同（需要不同的 per-bit DCA 延迟）、终端反射特征不同（需要不同的 ODT 值、不同的 DFE 系数）。一根统一的 MRW 全写同一个值——对 CLI 位置的芯片可能是最优的，对 DIMM 远端的芯片可能眼图已经闭合了。

**PDA（Per DRAM Addressability）** 就是打破这个"一刀切"限制的机制——让 Controller 能对同一 Rank 内的不同 DRAM 做独立配置。

---

## 4.16.1 DDR4 时代的 PDA 为什么不好用

DDR4 就已经有 PDA 了。它的思路很直接：发 MRW 命令时，用 DQ0 的电平来"选择"哪颗 DRAM 执行——DQ0 = 0 的 DRAM 执行，DQ0 = 1 的 DRAM 忽略。问题是这个"选择"在**每一次** PDA 操作时都需要 DQ 总线参与——慢。而且有死锁风险：如果某颗 DRAM 的 VrefDQ 偏差太大、已经无法正确采样 DQ0——那 Controller 就无法通过 PDA 去修复它的 VrefDQ（因为 PDA 本身依赖于 DQ0）。死循环。

DDR5 彻底重构了 PDA 的架构——把"一次性枚举"和"后续选择"分离。

---

## 4.16.2 枚举阶段：一次性给每颗芯片发"身份证"

系统初始化时（此时默认 VrefDQ 尚在可用范围内），Controller 通过 MPC 命令进入"PDA Enumerate Programming Mode"。在这个模式下，Controller 用 DQ0（x4/x8）或 DQL0（x16）上的电平，配合 DQS 翻转，为每颗 DRAM 分配一个唯一的 **4-bit PDA Enumerate ID**（0~15）。

具体操作是：Controller 在 DQ0 上驱动目标 DRAM 为 0、其他 DRAM 为 1 → 发 MPC PDA Enumerate ID 命令（OP[3:0] = 要分配的 ID 值）→ DQS 翻转让 DRAM 采样 DQ0 → DQ0 = 0 的那颗 DRAM 将自己 ID 设为 OP[3:0]。轮转完成 8~16 颗芯片后退出编程模式。

PDA 枚举期间 **DM 必须被禁用**（DM 和 PDA Enumerate 使用相同的 DQ/DQS 信号路径）。

---

## 4.16.3 选择阶段：CA 总线直接"点名"，不再需要 DQ

枚举完成后，每颗芯片有了自己的 ID。此后所有 PDA 操作（MRW、MPC、VrefCA、VrefCS）都通过 CA 总线上的 **PDA Select ID** 字段来指定目标芯片——**完全不再需要 DQ 总线参与选择**。

每颗 DRAM 将命令中的 Select ID 与自己的 Enumerate ID 比较：匹配就执行，不匹配就忽略。Select ID = 1111B 是"全选"——所有芯片都执行，等价于传统的非 PDA MRW。

---

## 4.16.4 实战：一个典型的 PDA 配置序列

假设 Rank 上有 8 颗 DRAM，已分配 ID 0~7：

```
// DRAM 3 离 Controller 最远，VrefDQ 需要偏高 2 步
PDA Select ID = 3 → MRW(MR10, VrefDQ = 0x42)

// DRAM 7 在 DIMM 最右端，ODT 需要更强终端以吸收反射
PDA Select ID = 7 → MPC Set RTT_PARK = 48Ω

// CL 是公共参数——所有芯片一样，用"全选"
PDA Select ID = 15 (1111B) → MRW(MR0, CL = 34)
```
